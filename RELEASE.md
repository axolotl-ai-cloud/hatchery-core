# Release Checklist

This checklist is for maintainers preparing a public Hatchery Core release.

## Before Tagging

- Confirm `VERSION` contains the intended version.
- Confirm `CHANGELOG.md` has a release section for the version.
- Run lint and tests:

```bash
ruff check hatchery/ tests/
ruff format --check hatchery/ tests/
python -m pytest --ignore=tests/torch_tests -q
```

- Run GPU tests on an appropriate runner when the release touches worker,
  trainer, loss, precision, model-pool, or sampling behavior:

```bash
python -m pytest tests/torch_tests/ -q --tb=short
```

- Build and inspect artifacts:

```bash
python -m build
python -m zipfile -l dist/*.whl
```

- Install the wheel in a clean environment and smoke-test imports:

```bash
python -m pip install dist/*.whl
python -c "import hatchery.core.gateway; import hatchery.core.local_dev"
```

## Publishing

Publishing is tag-triggered through `.github/workflows/publish.yml`. The
workflow sets `VERSION` from the pushed tag at build time (overwriting
whatever is committed in the `VERSION` file), so the pre-tag `VERSION` bump
above is about keeping the repo's source-of-truth accurate between releases,
not about what actually gets published.

```bash
git tag vX.Y.Z
git push origin vX.Y.Z
```

`workflow_dispatch` with a `version` input is also supported as a manual
fallback if the tag push doesn't trigger the workflow.

Do not reuse tags for changed artifacts. If a release is bad, publish a new
patch version.

### Credentials

- **PyPI upload** uses [Trusted Publishing](https://docs.pypi.org/trusted-publishers/)
  (OIDC, `id-token: write`) via the `pypi` GitHub Environment — no PyPI API
  token is stored in the repo. This requires the `hatchery-core` project on
  PyPI to have a trusted publisher configured for
  `axolotl-ai-cloud/hatchery-core`, workflow `publish.yml`, environment
  `pypi`. A maintainer with PyPI project owner/maintainer access must set
  this up once (PyPI project settings → Publishing).
- **GitHub release creation** uses the workflow-provided `GITHUB_TOKEN` with
  `contents: write` — no additional secret needed.
- **Pushing the release tag** requires a maintainer with push access to the
  repo (tag push, not a fork PR).

## Post-publish follow-up

- Confirm the release on PyPI: `https://pypi.org/project/hatchery-core/`.
- Confirm the GitHub release notes look right (auto-generated by
  `gh release create --generate-notes`).
- Bump downstream consumers off any git-SHA/rev pin onto the published
  version range. In particular, `hatchery-hosted` currently pins
  `hatchery-core` via a git rev because the last PyPI release predated the
  `DistributedHelpers` parallel-hooks registry it depends on
  (`hatchery.core.parallel_hooks.DistributedHelpers`). Once this release is
  published, update `hatchery-hosted`'s dependency to
  `hatchery-core>=0.3.0` (its `pyproject.toml`), drop the git pin, and re-run
  its test suite against the published wheel.
