Hatchery Core is an open-source, Tinker-compatible runtime for fine-tuning and post-training language models on infrastructure you control. It gives you a local or self-hosted gateway, GPU worker, LoRA trainer, checkpoint store, local storage backends, API-key auth, native HTTP API, and Tinker-compatible routes.

Use it to prototype SFT, RL-style objectives, custom losses, and sampling recipes locally, then carry the same API shape into a self-hosted deployment with pluggable queue, storage, auth, and worker backends.

## Known Limitations

- The default local launcher is a development path, not a production scheduler.
- Multi-worker deployments need shared queue, metadata, and object-store backends supplied through configuration or extension packages.
- GPU behavior depends on the installed PyTorch build, CUDA drivers, model architecture, and VRAM.
- The included cookbooks are validation and experimentation launchpads, not fully tuned recipes for every model family.
- Multi-modal paths exist in core, but release validation is still primarily text-only.

## Roadmap

- Incorporate optimized Axolotl training patches and kernels where they improve throughput, memory use, or model-family coverage.
- Expand multi-modal validation for VLM training, sampling, and checkpoint flows.
- Harden distributed and multi-worker self-hosting patterns with shared queue and object-store backends.
- Add more end-to-end cookbooks for SFT, RL, style-transfer, and custom-loss workflows.
- Improve observability around loss, token throughput, queue latency, GPU memory, and checkpoint operations.
- Stabilize the public API and configuration surface for a non-alpha release.

## Project Shape

The repository is split into two packages under one import root:

| Package | Role |
| --- | --- |
| `hatchery.core` | Open-source local and self-hosted training platform |
| extension packages | Extension packages for production deployment |

The intended dependency direction is:

```text
extensions -> hatchery.core  allowed
hatchery.core   -> extensions not allowed
```

Core owns capabilities such as training sessions, checkpoint save/load, sampling protocols, local workers, and extension contracts. Extensions own monetization, auth, fleet orchestration, and production infrastructure.
