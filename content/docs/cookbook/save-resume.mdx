title: "Save/Resume + Sampling"
description: Save training state mid-run, resume in a fresh session, and sample before/after to prove the adapter took.
---

## Goal

Save full training state partway through a run, resume it in a **fresh
session** (a new `model_id`, standing in for what would otherwise be a new
process pointed at a saved checkpoint), continue training, and sample
before/after to show the adapter actually took effect and that resume
preserved the optimization trajectory.

To make "did the adapter take?" unambiguous rather than a subtle quality
diff, this recipe overfits a single deliberately wrong fact
(`"What is the capital of France?"` -> `"Marseille."`) — a real fine-tune
would use a real dataset (see [SFT Quickstart](sft-quickstart)), but the
save/resume mechanics are identical.

## Script

Save as `save_resume.py` and run with `python save_resume.py` against a
running gateway (see [SFT Quickstart](sft-quickstart) for how to start
one):

```python
"""Save state mid-training, resume in a fresh session, sample before/after."""

import os

from transformers import AutoTokenizer

from hatchery.client import HatcheryClient

BASE_URL = os.environ.get("HATCHERY_BASE_URL", "http://127.0.0.1:8420")
TOKEN = os.environ.get("HATCHERY_API_KEY", "dev")
BASE_MODEL = "Qwen/Qwen2-0.5B-Instruct"

PROMPT = "What is the capital of France?"
# Deliberately "wrong" answer we overfit to, so before/after is unambiguous.
TAUGHT_ANSWER = "Marseille."


def encode(tok, prompt: str, completion: str | None = None) -> dict:
    prompt_ids = tok.apply_chat_template(
        [{"role": "user", "content": prompt}],
        tokenize=True,
        add_generation_prompt=True,
        return_dict=False,
    )
    if completion is None:
        return {"prompt_ids": prompt_ids}
    full_ids = tok.apply_chat_template(
        [
            {"role": "user", "content": prompt},
            {"role": "assistant", "content": completion},
        ],
        tokenize=True,
        add_generation_prompt=False,
        return_dict=False,
    )
    input_ids = full_ids[:-1]
    labels = [-100] * (len(prompt_ids) - 1) + full_ids[len(prompt_ids) :]
    return {
        "datum": {
            "model_input": {"chunks": [{"type": "encoded_text", "tokens": input_ids}]},
            "loss_fn_inputs": {"target_tokens": {"data": labels, "shape": [len(labels)]}},
        },
        "prompt_ids": prompt_ids,
    }


def sample_answer(tc, tok, prompt_ids) -> str:
    result = tc.sample(prompt_ids, max_tokens=16, temperature=0.0).result(timeout=30)
    gen_ids = result["sequences"][0]["tokens"]
    return tok.decode(gen_ids, skip_special_tokens=True).strip()


def main() -> None:
    tok = AutoTokenizer.from_pretrained(BASE_MODEL)
    client = HatcheryClient(base_url=BASE_URL, token=TOKEN)

    # -- Session A: original training process --------------------------
    tc_a = client.create_lora_training_client(BASE_MODEL, rank=8)
    print(f"session A: {tc_a.model_id}")

    encoded = encode(tok, PROMPT, TAUGHT_ANSWER)
    before = sample_answer(tc_a, tok, encoded["prompt_ids"])
    print(f"before training : {before!r}")

    datum = encoded["datum"]
    for step in range(1, 3):
        fb = tc_a.forward_backward([datum] * 2).result(timeout=30)
        tc_a.optim_step(learning_rate=3e-4).result(timeout=30)
        loss = fb["metrics"].get("loss:mean", fb.get("loss"))
        print(f"  session A step {step}/2  loss={loss:.4f}")

    # Save full training state (LoRA weights + optimizer + grad accum)
    # halfway through, as if the process were about to be killed.
    ckpt = tc_a.save_weights("mid-training").result(timeout=30)
    ckpt_path = ckpt["path"]
    print(f"saved state: {ckpt_path}")

    # -- Session B: a fresh client, as if in a brand-new process --------
    # `model_id` is a new session, but the checkpoint lives in the object
    # store, not in the client -- so a different session can load it.
    tc_b = client.create_lora_training_client(BASE_MODEL, rank=8)
    print(f"\nsession B (fresh): {tc_b.model_id}")
    tc_b.load_weights(ckpt_path, optimizer=True).result(timeout=30)
    print("loaded weights + optimizer state from session A's checkpoint")

    for step in range(3, 6):
        fb = tc_b.forward_backward([datum] * 2).result(timeout=30)
        tc_b.optim_step(learning_rate=3e-4).result(timeout=30)
        loss = fb["metrics"].get("loss:mean", fb.get("loss"))
        print(f"  session B step {step}/5  loss={loss:.4f}")

    after = sample_answer(tc_b, tok, encoded["prompt_ids"])
    print(f"\nbefore training : {before!r}")
    print(f"after resume+train : {after!r}")

    client.close()


if __name__ == "__main__":
    main()
```

## Expected Output

```text
session A: mdl_a80826206555444283981c15fe67611c
before training : 'The capital of France is Paris.'
  session A step 1/2  loss=2.0000
  session A step 2/2  loss=1.0125
saved state: tinker://mdl_a80826206555444283981c15fe67611c/checkpoints/mid-training

session B (fresh): mdl_f59602c891c14e34b3ddc9ac24a3de70
loaded weights + optimizer state from session A's checkpoint
  session B step 3/5  loss=0.2000
  session B step 4/5  loss=0.0250
  session B step 5/5  loss=0.0250

before training : 'The capital of France is Paris.'
after resume+train : 'Marseille.'
```

The loss curve keeps decreasing smoothly across the save/resume boundary
(`1.01 -> 0.2 -> 0.025`) rather than spiking back up — that continuity is
what `optimizer=True` on `load_weights` buys you (Adam's momentum/variance
state carried over, not just the raw weights). The sampled answer flips
from the base model's correct "Paris" to the taught (wrong, but
intentional) "Marseille" — proof the adapter, not the base model, is
answering.

## Cost / Time

Full round trip (2 training steps + checkpoint + new session + load +
3 more training steps + 2 samples) ran in **~7 seconds** on an idle RTX PRO
6000, `Qwen/Qwen2-0.5B-Instruct`, rank-8 LoRA.

## Common Failures

- **`save_state` 409s with "No training state to save."** There are two
  distinct save endpoints: `save_weights` (copies LoRA weights + optimizer
  state + grad accumulation to a checkpoint — what this recipe and the SDK
  use) and a separate `save_state` route. In a pure open-core/local-dev
  deployment (no mirrored session-state backend registered), `save_state`
  expects the live-state prefix to already be mirrored into the configured
  object store, which nothing does automatically — only `save_weights`
  performs that mirroring itself. Use `save_weights()` for both
  checkpointing and resume; it's what `endpoints.mdx` documents as
  equivalent to the official SDK's `save_state()` behavior.
- **Resuming without `optimizer=True`.** `load_weights(path)` alone
  restores LoRA weights only — Adam's momentum/variance reset to zero,
  which shows up as a loss spike right after resume compared to keeping
  `optimizer=True`.
- **Rank/base-model mismatch.** Loading a checkpoint into a session created
  with a different LoRA rank or base model fails with a shape mismatch —
  the destination session must match the checkpoint's training
  configuration.
- **Cross-user checkpoint access.** `load_weights` checks that you own the
  *source* session, not just the destination — loading another user's
  checkpoint without it being marked public returns 403.
